# 代码规范与可维护性说明

## 1. 项目分层

本项目采用较清晰的分层结构，便于在答辩中说明职责边界：

- `config`：集中管理运行目录、资源路径等配置。
- `domain`：使用 Java `record` 表达查询结果、统计结果和导入结果等数据对象。
- `repository`：封装 SQLite 表结构、入库、查询和聚合 SQL。
- `service`：组织导入、查询、分析、导出等业务流程。
- `ui`：负责 JavaFX 页面组装、图表展示、表格展示和后台任务调度。
- `util`：放置 GDELT 字段映射、行解析等通用工具。

这种结构可以避免 UI 直接处理数据库细节，也便于把数据导入、指标分析和结果导出流程讲清楚。

## 2. 命名与格式

代码整体遵循 Java 常见命名习惯：

- 类名使用大驼峰命名，例如 `GdeltImportService`、`DashboardSummary`。
- 方法和变量使用小驼峰命名，例如 `queryEvents`、`cooperationRankings`。
- 常量使用全大写下划线命名，例如 `DEFAULT_RANK_LIMIT`。
- 数据对象优先使用 `record`，减少样板 getter 和构造代码。

项目已增加 `.editorconfig`，统一 UTF-8 编码、缩进、换行和行尾空格规则，减少不同编辑器造成的格式差异。

## 3. 注释策略

注释以“解释职责和业务口径”为主，避免重复解释简单语句。重点补充了以下内容：

- UI 入口类说明页面组装和后台任务职责。
- Repository 说明 SQL 查询与聚合统计职责。
- 导入服务说明校验、解析、过滤、清洗、批量入库流程。
- 合作指数、风险指数和 K-Means 聚类说明计算口径适用于课程展示和相对比较。

对于答辩而言，核心指标可以概括为：合作指数综合合作占比、事件规模、Goldstein、AvgTone 和媒体关注度；风险指数综合冲突占比、冲突规模、负向 Goldstein 和负向 AvgTone。

## 4. 异常处理与稳定性

数据库访问和文件导入中的异常会转换为带上下文的运行时异常，方便界面展示错误原因。UI 下拉选项加载失败时采用降级策略，返回空列表以保证页面仍可打开，并允许用户手动输入国家代码。

自检入口类中的 `System.out.println` 保留用于命令行检查，不作为正式业务日志使用。

## 5. 质量检查

项目已增加轻量 Checkstyle 配置，优先检查：

- 文件编码。
- 制表符缩进。
- 超长行。
- 星号导入。
- 未使用导入。

该配置刻意保持宽松，目的是在不大规模重构现有课程项目的前提下，建立基础代码质量门禁。

## 6. 后续可优化方向

如果后续还有时间，可以继续做以下改进：

- 将 `MainView` 按页面拆分为多个 JavaFX 视图类。
- 将 `GdeltEventRepository` 按查询场景拆分为多个 Repository。
- 为 `GdeltLineParser`、导入过滤逻辑、合作/风险评分逻辑增加单元测试。
- 统一 JavaFX 后台任务创建方式，集中处理线程命名和异常提示。
