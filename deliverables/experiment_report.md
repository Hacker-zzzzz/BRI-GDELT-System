# 一带一路沿线国家合作态势分析系统实验报告

## 1. 实验背景

本项目为 Java 程序设计课程期末大作业，选题为“一带一路沿线国家合作态势分析系统”。老师给出的任务书要求学生围绕 GDELT 2.0 Event 数据完成一个 Java 窗口程序，系统应能够处理国际事件数据，并从合作态势、风险评估、区域差异和专题地图等角度进行分析展示。

GDELT 2.0 Event 数据记录了全球国家、组织和人物之间的事件交互，每条记录包含事件日期、参与方国家代码、事件编码、GoldsteinScale、NumMentions、AvgTone 和地理信息等字段。对于一带一路沿线国家而言，这些字段能够反映国家之间的合作、冲突、媒体关注度和情绪倾向。因此，本项目选择将 GDELT 原始事件数据转化为一个可查询、可统计、可展示、可导出的桌面系统。

本项目不是单纯的数据可视化页面，也不是只针对单个 CSV 文件的脚本，而是一个从数据导入到分析导出的完整 Java 应用。系统使用 JavaFX 构建桌面窗口，使用 SQLite 保存清洗后的事件数据，使用 Maven 统一管理依赖和构建过程，运行期文件统一放在 `D:\Temp\BRI-GDELT-System` 下，符合课程对窗口程序、数据库、固定目录和可提交性的要求。

## 2. 实验目的

本实验的总体目标是完成一个稳定、可运行、适合课堂答辩展示的 Java 桌面程序。具体目标如下。

1. 掌握 Java 面向对象程序设计方法，将系统拆分为 UI、Service、Repository、Domain、Util 等层次。
2. 掌握 JavaFX 桌面界面开发，完成导航、表格、图表、进度反馈和文件选择等交互。
3. 掌握 JDBC 和 SQLite 的基本使用，能够设计表结构、创建索引并完成批量写入和查询聚合。
4. 掌握 Java I/O 和压缩文件读取，支持 `.CSV`、`.tsv`、`.zip` 格式的 GDELT 数据导入。
5. 掌握异常数据处理方法，对坏行、空字段、数字格式错误和异常经纬度进行处理。
6. 掌握基本数据分析方法，构建合作指数、风险指数、区域汇总和 K-Means 聚类结果。
7. 掌握项目测试与交付整理方法，形成设计文档、实验报告和答辩 PPT 方案。

## 4. 需求分析

### 4.1 数据管理需求

系统需要支持 GDELT 2.0 Event 数据导入。GDELT 原始数据通常是无表头的 Tab 分隔文件，也可能以 `.zip` 压缩包形式提供。系统需要完成文件读取、字段解析、国家过滤、异常数据处理和数据库入库。

数据管理部分的重点不是简单读取文本，而是保证真实数据存在异常时程序仍然能够继续运行。因此系统对字段数量、日期格式、数字格式和经纬度范围都进行了检查，并将导入批次写入 `import_batches` 表，便于查看导入结果。

### 4.2 查询检索需求

系统需要提供多条件组合查询能力，包括日期范围、国家代码、事件类型和子区域。查询结果应能显示事件日期、Actor1/Actor2 国家代码、事件类型、GoldsteinScale、NumMentions、AvgTone 和来源文件等字段。

双边关系查询是任务书中的重点功能之一。系统支持以中国和某沿线国家为查询对象，统计两国之间的合作事件、冲突事件、平均 Goldstein、平均 AvgTone、媒体关注度和月度趋势，并展示明细事件。

### 4.3 挖掘分析需求

系统需要将原始事件转化为可解释的分析指标。根据任务书要求，项目实现了合作指数、风险指数、区域汇总和国家聚类。

合作指数主要反映一个国家与相关国家之间的合作活跃度和正向倾向；风险指数主要反映冲突事件、负向语调和潜在风险；区域汇总用于比较不同子区域的总体态势；国家聚类用于将沿线国家划分为不同类型，便于答辩展示和后续分析。

### 4.4 可视化与导出需求

系统需要使用窗口程序展示结果，而不能只输出命令行文本。因此项目使用 JavaFX 实现首页仪表盘、表格、折线图、柱状图、饼图和专题地图。结果导出部分支持 TXT、CSV 和 XLSX，便于继续整理实验报告和答辩材料。

## 5. 系统设计与实现

### 5.1 总体架构

系统采用分层结构。

```text
JavaFX UI 层
    -> Service 业务层
        -> Repository 数据访问层
            -> SQLite 数据库

Domain 模型层和 Util 工具层为各层提供支撑。
```

这种结构的好处是职责清楚：界面层负责展示和交互，业务层负责分析逻辑，数据访问层负责 SQL 和数据库操作，工具层负责 GDELT 字段映射和安全解析。相比把所有代码写在一个窗口类中，分层结构更容易维护，也更适合在答辩中说明设计思路。

### 5.2 核心类说明

| 类/模块 | 职责 |
|---|---|
| `BriGdeltApplication` | JavaFX 应用入口 |
| `StartupService` | 创建运行目录、初始化数据库、加载国家配置 |
| `AppPaths` | 统一维护运行期目录 |
| `MainView` | 构建主界面、导航、图表和表格 |
| `DatabaseManager` | 创建 SQLite 连接、初始化表和索引 |
| `GdeltImportService` | 导入 GDELT 文件、清洗、过滤、批量入库 |
| `GdeltEventRepository` | 事件写入、查询、统计和聚合 |
| `AnalysisService` | 合作指数、风险指数、区域分析和聚类 |
| `ReportExportService` | 导出 TXT、CSV 和 XLSX |
| `GdeltFieldMapper` | 维护 GDELT 字段下标 |
| `GdeltLineParser` | 安全解析 GDELT 数据行 |

### 5.3 数据库设计

数据库主要包含四张表。

| 表名 | 作用 | 说明 |
|---|---|---|
| `countries` | 国家配置表 | 保存 CAMEO 国家代码、ISO 代码、中文名、英文名、区域、经纬度等 |
| `gdelt_events` | 事件主表 | 保存清洗后的 GDELT 事件，是系统分析的核心数据来源 |
| `import_batches` | 导入批次表 | 记录导入文件、总行数、成功行数、跳过行数和状态 |
| `analysis_results` | 分析结果扩展表 | 为后续保存分析结果预留 |

事件表以 `global_event_id` 为主键，导入时使用 `INSERT OR IGNORE`。这样即使用户重复导入同一个 GDELT 文件，事件数量也不会重复累加。

为提高查询性能，系统为事件日期、Actor1、Actor2、事件类型、事件根编码、双边关系和地理点位建立索引。这些索引主要服务于日期筛选、国家筛选、双边关系、趋势分析和专题地图点位查询。

### 5.4 GDELT 字段解析

GDELT 2.0 Event 数据字段较多，且没有表头。本项目只提取课程任务所需的关键字段。

| 字段 | 项目内下标 | 含义 |
|---|---:|---|
| GlobalEventID | 0 | 事件唯一编号 |
| SQLDATE | 1 | 事件日期 |
| Actor1CountryCode | 7 | 主体 1 国家代码 |
| Actor2CountryCode | 17 | 主体 2 国家代码 |
| EventCode | 26 | 事件编码 |
| EventBaseCode | 27 | 事件基础编码 |
| EventRootCode | 28 | 事件根编码 |
| GoldsteinScale | 30 | 合作/冲突强度 |
| NumMentions | 31 | 媒体报道量 |
| AvgTone | 34 | 媒体语调 |
| ActionGeo_Lat | 56 | 事件纬度 |
| ActionGeo_Long | 57 | 事件经度 |

解析过程中，系统先检查字段数量是否足够，再逐项解析日期、整数和小数。如果字段缺失或格式错误，该行不会进入数据库，而是作为跳过行记录在导入结果中。经纬度如果超出合法范围，则置为空值，避免地图展示时出现异常点位。

### 5.5 数据导入流程

数据导入流程如下。

```text
选择 GDELT 文件
    -> 判断普通文本或 zip 压缩包
    -> UTF-8 读取
    -> Tab 分隔字段
    -> 安全解析关键字段
    -> 判断 Actor1/Actor2 是否命中国家配置
    -> 清洗经纬度
    -> 每 1000 条批量写入 SQLite
    -> 写入导入批次记录
```

导入服务保存了总行数、成功行数、跳过行数、实际插入行数和错误样例。这样做的好处是导入结果可解释：如果导入后事件数量没有明显增长，可以通过批次表判断是重复导入、国家过滤导致跳过，还是原始文件格式存在问题。

### 5.6 查询模块实现

查询模块支持多条件组合，包括起始日期、结束日期、任意参与国家、Actor1 国家、Actor2 国家、区域和事件类型。Repository 层根据条件动态拼接 `WHERE` 子句，并使用 PreparedStatement 绑定参数，避免直接拼接用户输入。

查询结果按日期和事件 ID 倒序排列，并限制最大返回数量。这样既能满足课堂展示时查看近期事件的需要，也能避免一次性加载过多数据导致界面卡顿。

### 5.7 双边关系实现

双边关系查询使用对称条件：

```text
(actor1_country_code = A AND actor2_country_code = B)
OR
(actor1_country_code = B AND actor2_country_code = A)
```

这样可以避免只统计单方向事件。双边统计结果包括事件总量、合作事件、冲突事件、其他事件、合作占比、冲突占比、平均 Goldstein、平均 AvgTone 和媒体关注度。月度趋势使用 `substr(event_date, 1, 7)` 按月份聚合。

### 5.8 合作指数实现

合作指数公式如下。

```text
cooperationIndex = clamp(
    100 * (
        cooperationRatio * 0.35
      + cooperationShare * 0.30
      + positiveGoldstein * 0.15
      + positiveTone * 0.08
      + mentionShare * 0.12
      - conflictRatio * 0.15
    )
)
```

该公式综合考虑合作占比、合作事件相对规模、正向 Goldstein、正向媒体语调和媒体关注度，同时对冲突占比进行扣分。`cooperationShare` 和 `mentionShare` 使用全体国家中的最大值归一化，公式中的 `clamp` 将结果控制在 0 到 100 之间，便于页面显示和排名比较。

该模型的设计重点是可解释性。答辩时可以直接说明：合作占比越高、相对合作规模越大、Goldstein 越正、AvgTone 越正、媒体关注度越高，合作指数越高；冲突占比越高则会降低合作指数。

### 5.9 风险指数实现

风险指数公式如下。

```text
riskIndex = clamp(
    100 * (
        conflictRatio * 0.35
      + conflictShare * 0.45
      + negativeGoldstein * 0.12
      + negativeTone * 0.08
    )
)
```

其中 `conflictShare` 使用全体国家中的最大冲突事件量归一化，避免事件绝对数量过大时所有高位国家都被截断为 100 分。

风险等级划分为低、中、高、极高四档：

| 分数区间 | 风险等级 |
|---|---|
| 0-25 | 低 |
| 25-50 | 中 |
| 50-75 | 高 |
| 75-100 | 极高 |

风险指数并不直接等同于现实投资结论，而是根据 GDELT 事件数据给出的量化参考。报告和答辩中需要说明该指标的使用边界，避免过度解释。

### 5.10 区域分析与国家聚类

区域分析按国家配置中的 `region` 字段聚合，统计区域国家数、事件总量、合作事件、冲突事件、平均 Goldstein、平均 AvgTone、媒体关注度、区域合作指数和区域风险指数。

国家聚类采用项目内轻量 K-Means，`k=4`。输入特征包括合作指数、风险指数、冲突占比、Goldstein、AvgTone 和事件量归一化值。聚类标签固定解释为：

1. 深度合作伙伴。
2. 稳定合作。
3. 存在风险。
4. 高度紧张。

由于课程项目重点是 Java 综合应用和可解释分析，本项目没有强依赖 Weka，而是在项目内部实现了可控、可说明的轻量聚类流程。

### 5.11 界面实现

系统界面采用顶部标题栏、左侧导航栏和中央内容区。主要页面包括：首页仪表盘、数据导入、事件查询、双边关系、合作态势、风险评估、区域分析、国家聚类、专题地图和结果导出。

导入、查询和分析等耗时任务使用 JavaFX 后台任务执行，避免主线程阻塞。表格和图表用于展示核心数据，简短研判文字用于帮助用户理解指数含义。

### 5.12 结果导出实现

系统导出四类文件：

1. `bri_gdelt_snapshot_*.txt`：总体概况和重点研判。
2. `cooperation_rankings_*.csv`：合作指数排名。
3. `risk_rankings_*.csv`：风险指数排名。
4. `bri_gdelt_analysis_*.xlsx`：包含合作排名、风险排名、区域汇总、国家聚类四个工作表。

导出功能说明系统不仅能在界面中展示结果，也能把分析结果保存到文件中，用于报告整理和后续分析。

## 8. 遇到的问题与解决方案

### 8.1 GDELT 字段多且无表头

问题：GDELT 原始文件字段多，且没有表头。如果直接按字符串下标读取，很容易出现下标混乱。

解决方案：单独编写 `GdeltFieldMapper`，将项目需要的字段下标集中维护，并在注释中说明 GDELT 文档通常从 1 开始编号，而 Java 数组下标从 0 开始。

### 8.2 原始数据存在坏行

问题：真实数据中可能出现字段不足、日期错误、数字格式错误等情况。

解决方案：使用 `GdeltLineParser.parseDetailed` 返回解析结果。解析失败时跳过该行并记录错误原因，而不是抛出异常导致整个导入失败。

### 8.3 重复导入导致数据重复

问题：用户可能重复导入同一天的 GDELT 文件，如果没有去重，事件数量会被放大。

解决方案：事件表以 `global_event_id` 为主键，入库使用 `INSERT OR IGNORE`。这样重复导入不会重复计数。

### 8.4 大数据量查询性能

问题：事件数量较大时，按日期、国家和事件类型查询可能变慢。

解决方案：为日期、Actor1、Actor2、事件类型、事件根编码和双边关系建立索引。同时查询结果设置返回数量上限，避免一次性加载过多表格行。

### 8.5 UI 响应问题

问题：导入和查询都可能耗时，如果直接在 JavaFX 主线程执行，界面会卡顿。

解决方案：耗时任务使用 JavaFX `Task` 在后台执行，执行完成后再更新界面。这样能保持窗口可响应。

### 8.6 分析模型可信度

问题：合作指数和风险指数是项目自定义模型，如果公式过于复杂，答辩时难以解释；如果过于简单，又缺乏分析意义。

解决方案：指标只使用任务书明确提到的字段：事件量、Goldstein、AvgTone、NumMentions 和冲突占比。公式保持简洁，并明确说明指标是课堂项目中的量化参考，不夸大为现实预测模型。

## 9. 不足与改进

1. 聚类部分采用项目内轻量 K-Means，未引入 Weka。优点是依赖少、运行稳定；不足是模型评价指标不够丰富。后续可加入轮廓系数或与 Weka 聚类结果对比。
2. 地图部分已经能展示事件经纬度点位，但底图精度和交互能力仍有提升空间。后续可加入更精细的 GeoJSON 底图和区域着色。
3. 当前导出以 TXT、CSV 和 XLSX 为主，尚未自动生成图文并茂的 PDF/HTML 年度报告。后续可把年度报告作为选做功能扩展。
4. 国家配置数量为 145，后续可根据老师认可的沿线国家清单继续补全到 150+。
5. 当前指标权重主要依据可解释性和课堂展示设计，后续可根据专家打分、历史样本或归一化方法进一步优化。
